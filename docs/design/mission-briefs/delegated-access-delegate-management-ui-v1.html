Landlord-facing delegated access management without shared logins.

Mission Goal

Create the first owner-facing UI for landlord delegates so owners can view pending invitations, active/revoked grants, invite delegates, and revoke access using the merged backend contracts.

Scope

  • Delegate Management page or Settings subpage for landlord owners.
  • List delegates, invitations, and grants with clear statuses.
  • Invite Delegate workflow using existing invitation API.
  • Revoke Access action using owner-management API.
  • Role and scope display suitable for v1 landlord review.
  • Calm empty state and security messaging against shared logins.

Non-goals

  • No email dispatch or notification sending.
  • No invite acceptance UI unless already safe and minimal.
  • No property manager companies or contractor organizations.
  • No billing delegation or custom roles.
  • No security rule changes or backend redesign.
  • No exposure of internal IDs, tokens, token hashes, or sensitive metadata.

Key Principles

O
Owner-first: only landlord owners can manage delegate access.
F
Fail-closed: ambiguous entitlement or auth state blocks management actions.
S
No shared logins: delegates must use their own accounts.
R
Revocation clarity: access removal must feel immediate and understandable.
P
Projection-safe UI: show human-readable role/scope/status without raw secrets.

Security / Privacy Warning

Primary copy: Never share your login. Invite delegates to their own account. The UI should explain that revoking access prevents future delegated actions while preserving audit history.

UX Flow

  • Owner opens Delegate Management from Settings or landlord workspace navigation.
  • Owner sees overview: pending, active, revoked, expired.
  • Owner invites a delegate by email, predefined role, property scope, workspace scope.
  • Owner reviews delegates/grants and revokes an active grant when needed.
  • Empty state explains delegated access without implying email dispatch.

Acceptance Criteria

  • Owner can view delegates, grants, and invitations without duplicate inbox/workspace concepts.
  • Non-owner or ambiguous auth state cannot access the management UI.
  • Owner can create a pending invitation with valid role/scope.
  • Owner can revoke an active grant and see the revoked state immediately.
  • Empty, loading, error, and mobile states are intentional.

Revocation Behavior

  • Only active grants can be revoked.
  • Grant record is preserved, not deleted.
  • UI updates grant status to Revoked immediately after success.
  • Copy clarifies revocation affects future delegated access.

Risks

  • Owner-only UI guard must not rely on client assumptions alone.
  • Role/scope labels must not expose internal implementation details.
  • No token/hash data should appear in logs, UI, tests, or errors.
  • Email expectations must be clear because dispatch is out of scope.

QA Checklist

  • Owner route loads and shows security message.
  • Invite form validates role, email, property scope, workspace scope.
  • Delegates, grants, and invitations render correct status labels.
  • Revoke action works and cannot be repeated as active revoke.
  • Mobile layout has no horizontal overflow.

Validation Requirements

  • Frontend tests for page, invite workflow, revoke workflow, and owner-only states.
  • Backend tests only if a small API adapter change requires them.
  • Frontend build.
  • git diff --check.
  • Manual preview QA before merge.

Merge Readiness Criteria

  • Mission Brief reviewed and accepted before implementation.
  • Implemented scope matches this brief.
  • Checks are green and manual preview QA passes or findings are triaged.
  • No backend redesign, email dispatch, billing delegation, custom roles, or org hierarchy added.

Audit Events

Existing backend audit events are consumed as contract behavior. UI should not create client-side audit records.
