Mission Brief: Property Manager Company Management UI V1

Design the first owner/admin interface for landlord-to-property-manager-company relationships and staff assignments. This brief intentionally stops before implementation because the merged backend APIs are not yet sufficient for a safe, complete V1 UI without additional read/discovery endpoints.

1. Mission Goal

Introduce a governed UI layer where landlord owners can review PM company relationships and where PM Company Owners/Admins can accept relationships and manage staff assignments, while preserving landlord data ownership, company consent, assignment scope ceilings, and fail-closed permission boundaries.

2. Scope

Landlord Side
  • View pending, active, suspended, and terminated PM company relationships.
  • Create pending relationships when the PM company can be selected by safe label.
  • Suspend, reactivate, or terminate relationships with confirmation.
  • Show relationship scope and clear pending acceptance copy.
Company Side
  • Company Owner/Admin views pending and active relationships for their company.
  • Accept pending relationships without modifying landlord-approved scope.
  • View and manage staff assignments for active relationships.
  • Use predefined assignment role templates only.

3. Non-Goals

No contractor organizations No billing delegation No custom permissions No public branding profile No tenant-facing PM UI No emails No migrations/backfills No dashboard redesign

4. User Roles

Role V1 UI Authority
Landlord Owner Creates pending relationships, sees landlord-scoped relationships, suspends/reactivates/terminates.
PM Company Owner/Admin Accepts pending relationships and manages company staff assignments within approved scope.
Regional Manager / Staff Roles May receive assignments later; no staff management or relationship acceptance authority in V1.

5. Landlord UX Flow

  • Open PM Company Management from the landlord account/workspace shell.
  • Review relationship cards grouped by pending, active, suspended, and terminated.
  • Create a pending relationship by selecting a PM company safe label and relationship scope.
  • See copy: Access becomes active only after the PM company accepts.
  • Confirm before suspend, reactivate, or terminate; termination must clearly preserve history.

6. PM Company Admin UX Flow

  • Open PM Company Management from a company-scoped account surface.
  • Select one of the actor's PM companies by safe label if they belong to multiple companies.
  • Review pending relationships and accept without altering landlord-approved scope.
  • Review active relationships before creating staff assignments.
  • See suspended/terminated relationships as blocked, preserved history.

7. Staff Assignment UX Flow

  • Company Owner/Admin selects an active landlord-company relationship.
  • Select active company staff from a safe member list, not by raw user ID.
  • Choose role template: Regional Manager, Property Manager, Leasing Agent, Office Administrator, Maintenance Coordinator, or Read-only Staff.
  • Choose scope that narrows, never broadens, the relationship scope.
  • Confirm before suspend, reactivate, or remove; removed assignments remain visible as history.

8. API Endpoints Used

Surface Current Endpoint Readiness
Landlord relationship list/create/lifecycle /api/landlord/property-manager-company-relationships Usable for landlord list/lifecycle; create needs safe company discovery first.
Company relationship acceptance /api/property-manager-companies/:companyId/relationships/:relationshipId/accept Accept works when IDs are already known; missing company-side relationship list.
Company staff assignments /api/property-manager-companies/:companyId/staff-assignments Lifecycle works when company, relationship, and staff user IDs are known; missing staff list.

API Sufficiency Finding

Existing APIs are not sufficient for a safe complete V1 management UI. A UI would currently need hidden or manually entered internal IDs for PM companies, company relationships, and staff users. That would violate the no-raw-ID UX boundary and produce an unusable company admin workflow.

Recommended prerequisite backend mission: feat/property-manager-company-management-read-api-v1.

Missing Backend Support

  • Company discovery/selection by safe label for landlord relationship creation.
  • Authenticated company context: list companies the actor belongs to with role/status.
  • Company-scoped relationship list for pending/active/suspended/terminated records.
  • Safe active staff membership list for assignment creation.
  • Optional landlord-visible staff assignment list for relationship transparency.
  • Safe projections that include labels, statuses, scopes, and timestamps without raw IDs as labels.

9. State/Status Display

  • Relationship: Pending, Active, Suspended, Terminated.
  • Assignment: Active, Suspended, Removed.
  • Pending relationship copy must emphasize PM company acceptance is required.
  • Suspended blocks access temporarily; terminated/removed preserve history and block access.

10. Safety / Permission Warnings

  • Landlord owner, PM company, company staff, and assignments must be visually distinct.
  • No shared logins, staff-to-staff delegation, chain delegation, or custom permission builder.
  • Billing/settings remain landlord-owner only and must never be selectable.
  • Relationship scope is the ceiling; assignment scope can only narrow it.
  • Confirm suspend, terminate, and remove before API mutation.

11. Mobile Expectations

  • Default to compact grouped sections with counts and clear state labels.
  • Relationship and assignment actions should move into menus or bottom-safe confirmation panels.
  • Long company/staff labels must wrap or truncate without overlapping state badges.
  • History sections may be collapsible but must not imply deletion.

12. Risks

  • Current route set lacks safe discovery APIs, making full UI implementation premature.
  • Company-side navigation/auth destination is not yet modeled in frontend routing.
  • Staff assignment creation without staff membership list would force unsafe user ID entry.
  • Landlord visibility into staff assignments may require a landlord-scoped safe projection endpoint.

13. Acceptance Criteria

  • Mission Brief accepted before any UI implementation starts.
  • If backend read APIs remain incomplete, open the backend read/discovery mission first.
  • If UI proceeds later, no raw company/user/landlord/property IDs appear as labels.
  • All mutations require confirmation and preserve backend lifecycle semantics.

14. QA Checklist

  • Verify landlord relationship list and lifecycle actions with safe labels.
  • Verify company admin can list and accept pending relationships.
  • Verify staff assignment create/list/suspend/reactivate/remove with scope ceiling enforcement.
  • Verify non-admin staff cannot manage relationships or assignments.
  • Verify mobile grouped sections, confirmations, empty states, and no text overlap.

15. Merge Readiness Criteria

  • Brief documents API insufficiency and recommended backend prerequisite.
  • No implementation, backend, schema, auth, billing, contractor, or frontend runtime change in this PR.
  • git diff --check passes.
  • Next mission is explicitly reviewed before UI implementation begins.
