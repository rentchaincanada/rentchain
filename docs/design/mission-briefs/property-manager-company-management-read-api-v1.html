Mission Brief: Property Manager Company Management Read API V1

Add safe read-only backend APIs required before the PM Company Management UI can be implemented. This mission fills discovery and projection gaps only; it must not add write behavior, UI routes, emails, billing authority, contractor organization behavior, or custom permissions.

1. Mission Goal

Provide server-authorized, projection-safe read surfaces for landlord owners and PM Company Owners/Admins so the next UI mission can render company relationships, company context, staff membership, and assignment state without raw-ID entry or cross-tenant/company leakage.

2. Scope

Landlord Reads
  • Safe PM company discovery/lookup for relationship creation.
  • Landlord-scoped relationship list with safe company label and relationship scope.
  • Landlord-visible staff assignment summaries under landlord-owned relationships.
Company Reads
  • Actor company context list for active Company Owner/Admin memberships.
  • Company-scoped relationship list with safe landlord workspace label.
  • Company member list and assignment projections for management UI.

3. Required Read Surfaces

Surface Expected Projection Authority
Company discovery / lookup Company reference token/id for API use, safe display label, status if needed. Authenticated landlord owner; no private company metadata.
Landlord relationship listing Company label, relationship status, scope, lifecycle timestamps, safe assignment summary. Landlord scope from auth context only.
Company relationship listing Safe landlord workspace label, status, scope, pending acceptance marker, timestamps. Active Company Owner/Admin membership for that company.
Company membership/staff listing Safe staff label, role, status, lifecycle timestamps needed by UI. Active Company Owner/Admin membership for that company.
Staff assignment projections Staff label, role, status, scope, relationship reference, lifecycle timestamps. Company Owner/Admin or landlord owner scoped to the relationship.

4. Proposed API Shape

  • GET /api/landlord/property-manager-companies/search?q=...
  • GET /api/landlord/property-manager-company-relationships with enriched safe summaries.
  • GET /api/landlord/property-manager-company-relationships/:relationshipId/staff-assignments
  • GET /api/property-manager-companies/my-companies
  • GET /api/property-manager-companies/:companyId/relationships
  • GET /api/property-manager-companies/:companyId/members
  • GET /api/property-manager-companies/:companyId/staff-assignments?relationshipId=... safe projection hardening.

Exact route names may follow local conventions during implementation, but the authority and projection contracts above are required.

5. Projection Rules

  • Raw IDs may be returned only as opaque API references needed for follow-up API calls, never as labels.
  • User-facing labels must use safe company, landlord workspace, and staff display labels.
  • Do not expose audit event IDs, internal membership IDs as labels, token-like fields, private metadata, billing data, or unrelated PII.
  • Assignment summaries must preserve role, status, scope, and timestamps without widening visibility.

6. Security Model

  • Landlord reads derive landlord scope from authenticated landlord owner context only.
  • Company reads require active Company Owner/Admin membership for the requested company.
  • Query/body cannot override landlord or company authority.
  • Cross-landlord and cross-company records are hidden or denied fail-closed.
  • Malformed company, membership, relationship, or assignment records fail closed.
  • Billing/settings remain out of scope and must not be included in management projections.

7. Non-Goals

No UI No dashboard No create/update/delete behavior No emails No contractor organizations No billing delegation No custom permissions No migrations/backfills

8. Existing API Gap

Current write/lifecycle APIs require known company, relationship, and staff user identifiers. The UI cannot safely collect those identifiers from humans. This mission adds the missing read model layer that lets the UI select safe labels while still sending canonical API references server-side.

9. Implementation Plan

  • Add backend read service helpers near existing PM company relationship/assignment services.
  • Reuse existing Firestore collection constants and projection-safe patterns.
  • Add landlord routes under the existing landlord PM company route namespace.
  • Add company routes under /api/property-manager-companies.
  • Keep existing write APIs unchanged except for projection hardening if needed.

10. Tests

  • Company discovery returns only safe active company projections.
  • Landlord relationship list is auth-landlord scoped and includes safe assignment summary.
  • Company relationship list requires active Company Owner/Admin and denies non-admin roles.
  • Company member list includes active, suspended, and removed members with safe staff labels, role, status, and no private metadata.
  • Landlord and company assignment projections deny cross-landlord/cross-company reads.
  • Malformed records and unknown roles/statuses fail closed.

11. QA / Validation

  • Focused backend route/service tests for each read surface.
  • Cross-landlord and cross-company denial tests.
  • Safe projection tests for no audit IDs, no raw IDs as labels, no private metadata leakage.
  • Backend build.
  • git diff --check.
  • No frontend build required unless implementation unexpectedly touches frontend, which is out of scope.

12. Risks

  • Company discovery could overexpose private company data if not tightly projected.
  • Staff labels may be limited until user profile display-name data is available; email-safe labels may be needed.
  • Landlord-visible staff assignment summaries must not become staff management authority.
  • Relationship assignment summaries must not imply billing/settings access.

13. Acceptance Criteria

  • All required read surfaces exist and are covered by backend tests.
  • Landlord reads cannot escape authenticated landlord scope.
  • Company reads cannot escape active Company Owner/Admin company scope.
  • Safe projections include labels/status/scope/timestamps and exclude private metadata.
  • No write behavior, UI, emails, billing delegation, contractor orgs, custom permissions, or migrations are added.

14. Merge Readiness Criteria

  • Mission Brief reviewed and accepted before implementation.
  • Focused backend tests pass.
  • Backend build passes.
  • git diff --check passes.
  • Contract review confirms safe projections and no scope creep.
  • PR remains draft until checks and contract review are complete.

15. Next Mission Unblocked

After this read API lands, the UI mission feat/property-manager-company-management-ui-v1 can proceed with safe selectable company, relationship, member, and assignment data.
