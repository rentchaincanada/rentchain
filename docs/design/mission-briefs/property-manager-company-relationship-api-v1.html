Next Mission: feat/property-manager-company-relationship-api-v1

Backend/API-only lifecycle endpoints for landlord-to-property-manager-company relationships, built on the merged Property Manager Company foundations. The mission keeps landlord ownership authoritative, fails closed on ambiguous scope, requires mutual acceptance before activation, and records metadata-only audit events for every lifecycle action.

Mission Goal

Expose backend API foundations that let landlord owners list, create, suspend, reactivate, and terminate Property Manager Company relationships, while modeling activation as a PM Company Admin acceptance step. New relationships must start pending and must not grant company authority until the company side accepts.

Ownership And Acceptance Model

Landlord relationship authority originates from the landlord side, but activation requires mutual acceptance: Landlord Owner creates the relationship, the record starts pending, and a PM Company Admin acceptance action moves it to active. The PM company cannot self-grant access, and the landlord remains the data owner.

Proposed API Surface

Action Endpoint Shape Contract
List GET /api/landlord/property-manager-company-relationships Return only relationships for the authenticated landlord owner context.
Create POST /api/landlord/property-manager-company-relationships Create a preserved relationship record in pending status. Do not create active relationships by default.
Accept / Activate POST /api/property-manager-company/relationships/:relationshipId/accept Move a pending relationship to active only when an authorized PM Company Admin accepts. If this route is deferred, the service model must still assume mutual acceptance.
Suspend POST /api/landlord/property-manager-company-relationships/:relationshipId/suspend Temporarily block access while preserving relationship and audit history.
Reactivate POST /api/landlord/property-manager-company-relationships/:relationshipId/reactivate Return a suspended relationship to active when scope and ownership still validate.
Terminate POST /api/landlord/property-manager-company-relationships/:relationshipId/terminate Block access, preserve history, and store termination metadata. No hard delete.

Status Behavior

Transition Allowed Meaning
pending -> active Yes, by PM Company Admin acceptance Company consent is recorded before scoped authority can exist.
active -> suspended Yes, by landlord owner Temporarily blocks access while preserving relationship history.
suspended -> active Yes, by landlord owner reactivation Restores an existing accepted relationship when scope and ownership still validate.
active -> terminated Yes, by landlord owner Ends the relationship, blocks access, and preserves audit/history.
suspended -> terminated Yes, by landlord owner Ends a paused relationship without reactivating it first.
pending -> terminated Yes, by landlord owner Stops a proposed relationship before company acceptance without deleting history.
pending -> suspended No in V1 Use termination semantics for a pending relationship that should not proceed.
Other transitions No terminated -> active, terminated -> suspended, repeated same-status mutations, unknown statuses, and malformed records fail closed.

Audit Contract

Each lifecycle action creates a metadata-only event:

relationship_created relationship_activated relationship_suspended relationship_reactivated relationship_terminated

Event metadata must preserve actor user, property manager company, landlord context, relationship, status transition, property and workspace scope, target resource, outcome, timestamp, and reason where applicable. Audit payloads must not contain secrets, raw provider payloads, or user-facing raw ID labels.

Security Model

  • Landlord-side relationship management is landlord-owner only.
  • PM company activation requires an authorized PM Company Admin acceptance action.
  • Landlord scope resolves from authenticated server context, not query or body input.
  • Cross-landlord relationship reads or mutations are denied.
  • Property Manager Company existence and active company status are validated before creation.
  • Suspended or terminated companies cannot receive new active relationships.
  • Property and workspace scopes are validated before create or status mutation.
  • Relationship scope supports all current properties, selected properties, and workspace scope.
  • Relationship scope is a ceiling; future staff assignments cannot exceed it.
  • Invalid status transitions are rejected and audited as failures where appropriate.
  • Billing and settings authority remains landlord-owner only.
  • Malformed persisted records, unknown statuses, and unsupported scopes fail closed.
  • Responses use safe labels and projections; raw internal IDs are not user-facing labels.

Future Compatibility

  • One landlord can hold relationships with multiple PM companies.
  • One PM company can serve many landlords through separate landlord-approved relationships.
  • The relationship model must support future staff assignments without changing ownership semantics.
  • The future PM company console must consume accepted relationships, not self-created grants.

Implementation Boundaries

Backend/API only. No frontend routes, no landlord or company dashboard, no invite emails, no staff assignment UI, no contractor organizations, no billing delegation, no custom permissions, and no migration or backfill work.
