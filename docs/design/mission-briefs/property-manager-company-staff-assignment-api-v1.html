Next Mission: feat/property-manager-company-staff-assignment-api-v1

Backend/API-only routes and services for Company Owner/Admin staff assignment management. This mission exposes controlled create, list, suspend, reactivate, and remove operations on top of the merged staff assignment foundations, while preserving scope ceilings, metadata-only audit events, and fail-closed authority.

Mission Goal

Add company-scoped backend APIs that let active Company Owners/Admins manage PM company staff assignments for active landlord-company relationships. Assignment APIs must preserve the distinction between company membership and landlord relationship authority: membership alone grants no landlord access, and assignment only narrows authority already granted by an active relationship.

Required APIs

Operation Contract
Create assignment Company Owner/Admin creates an active assignment under an active landlord-company relationship.
List assignments Company Owner/Admin gets company-scoped list, optionally relationship-scoped, using safe projections only.
Suspend assignment active -> suspended; access blocked temporarily and audit preserved.
Reactivate assignment suspended -> active; only if relationship, membership, and scope remain valid.
Remove assignment active/suspended -> removed; no hard delete, history preserved.

Route Shape

Proposed mount: /api/property-manager-companies/:companyId/staff-assignments, with relationship-specific filtering via path or query where practical. All company and relationship authority must resolve server-side; request body/query data cannot override company, landlord, relationship, or scope ownership.

Create Contract

  • Authenticated actor is required.
  • Actor must have active Company Owner/Admin membership for :companyId.
  • Target staff user must have active membership in the same PM company.
  • Relationship must exist, belong to the same PM company, and be active.
  • Staff role must be one predefined assignment template.
  • Assignment property/workspace scope must be within the relationship ceiling.
  • Billing/settings scope remains denied.
  • Assignment creation cannot broaden landlord-approved authority or activate access by itself.

Lifecycle Contract

Valid Invalid
active -> suspended removed -> active
suspended -> active removed -> suspended
active -> removed removed -> removed
suspended -> removed suspended -> suspended and active -> active

Audit Contract

Every lifecycle action must append metadata-only audit events:

staff_assignment_created staff_assignment_suspended staff_assignment_reactivated staff_assignment_removed

Metadata must include actorUserId, actorCompanyId, actingForLandlordId, relationshipId, assignmentId, staffUserId, role/template, scope, transition, outcome, timestamp, and reason where applicable. Audit payloads must not expose secrets, provider payloads, or user-facing raw ID labels.

Security Model

  • Company Owner/Admin only for create, list, suspend, reactivate, and remove.
  • Regional Manager, Property Manager, Leasing Agent, Office Administrator, Maintenance Coordinator, Read-only Staff, and unknown roles are denied.
  • Inactive actor memberships and inactive target staff memberships are denied.
  • Company-scoped only; no cross-company assignment access.
  • No cross-landlord access or body/query landlord override.
  • Pending, suspended, terminated, missing, or malformed relationships fail closed.
  • Malformed assignment, membership, relationship, role, status, or scope fails closed.
  • Assignment scope may narrow but never broaden relationship scope.
  • settings_billing remains denied in V1.
  • Explicit deny wins in service/evaluator flows.
  • Safe projections must omit audit internals and avoid raw IDs as user-facing labels.
