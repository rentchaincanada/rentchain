Backend email delivery for delegated-access invitation acceptance.

Mission Goal

Complete the delegated-access invitation delivery step so owner-created invitations send a secure acceptance email and support safe resend behavior without exposing raw tokens through APIs or logs.

Scope

  • Send invitation email after successful delegated-access invitation creation.
  • Use existing approved email provider and service patterns.
  • Generate acceptance URL containing the one-time raw token.
  • Add narrow resend behavior if it fits existing lifecycle safely.
  • Record metadata-only audit events for dispatch and resend outcomes.
  • Keep API list responses free of tokens and token hashes.

Non-goals

  • No property manager company model.
  • No contractor organization model.
  • No billing delegation or custom roles.
  • No delegate dashboard redesign.
  • No marketing email behavior.
  • No scheduling integration.
  • No security rule changes unless absolutely required.

Key Principles

1
Owner-controlled: invitation delivery happens only from owner-authorized invitation creation or resend.
2
Token-minimal: raw token appears only inside the generated acceptance URL.
3
Fail closed: expired, cancelled, accepted, or invalid invitations cannot be accepted or resent unsafely.
4
Audit safe: audit records contain delivery metadata, never raw tokens, token hashes, or provider payloads.
5
Own account: email wording reinforces that delegates must use their own account, not shared landlord credentials.

Email Flow

InviteOwner creates scoped invitation.
DispatchEmail sent with acceptance URL.
AcceptDelegate authenticates and accepts token.
ActiveGrant becomes active with role and scope.
RevokeOwner revokes future access.

API Endpoints / Changes

Method Endpoint Expected behavior
POST /api/landlord/delegated-access/invitations Create pending invitation and dispatch email after successful record creation.
POST /api/landlord/delegated-access/invitations/:id/resend Optional narrow resend for pending, unexpired invitations only.
POST /api/delegated-access/invitations/accept Existing acceptance flow continues to enforce lifecycle and token hash checks.

Acceptance Link Behavior

  • Email contains acceptance URL with raw token as the only raw-token surface.
  • Server stores and compares token hash only.
  • List, management, and audit APIs never return raw token or token hash.
  • Invalid tokens return safe errors without revealing invitation existence.

Expiration / Resend / Cancel

  • Expired invitations cannot be accepted or resent as usable links.
  • Cancelled invitations cannot be accepted or resent.
  • Accepted invitations cannot be reused or resent.
  • Resend is owner-only, pending-only, and cannot create duplicate active grants.
  • Resend preserves or safely refreshes delivery metadata without changing role or scope.
  • Cancel remains owner-only and withdraws acceptance eligibility immediately.

Email Content

  • Who invited the delegate.
  • Assigned delegated role.
  • Workspace and property scope when safe to show.
  • Expiration time or date.
  • Clear warning: use your own account; never share landlord credentials.

Audit Events

  • delegate.invitation.email_dispatched
  • delegate.invitation.email_resend_requested
  • delegate.invitation.email_dispatch_failed
  • Include actorUserId, actingForLandlordId, invitationId, outcome, provider key if safe, and timestamp.
  • Exclude raw token, token hash, full provider response, and sensitive message payload.

Risks / Failure States

  • Email provider unavailable or misconfigured.
  • Dispatch succeeds but provider later bounces.
  • Failed dispatch must not be shown or stored as delivered.
  • Invitation may remain pending with clear dispatch_failed or resend_available state.
  • Raw token leaks through logs, responses, or audit metadata.
  • Resend accidentally revives cancelled or expired invitations.
  • Cross-landlord invitation metadata leaks through resend or errors.

Acceptance Criteria

  • Creating a valid invitation sends one delegated-access invitation email.
  • Email includes a working acceptance URL and no unrelated marketing content.
  • Raw token is not returned by create, list, owner-management, or audit responses.
  • Pending unexpired invites may be resent if implemented; other states fail closed.
  • Cancelled, expired, and accepted invitations remain non-accepting.
  • Email failure leaves invitation lifecycle safe and visibly retryable, not falsely delivered.

QA Checklist

  • Create invite in test mode and verify email delivery or mocked dispatch.
  • Open acceptance link and confirm pending invite can be accepted.
  • Confirm cancelled, expired, and accepted links fail closed.
  • Confirm resend behavior and audit event if resend is included.
  • Confirm no token appears in API responses, logs, or audit metadata.
